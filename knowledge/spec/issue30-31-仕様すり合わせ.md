## フィードバック仕様（Issue #31）

### 概要

おさわりのたびに**必ず**再生されるヒロインの反応。

```
プレイヤー 胸さわる
  → ヒロイン「っふ……！」（アニメ：shoulder_twitch）
  → パラメータ+
```

### フィードバックとシナリオの違い

| 項目               | フィードバック             | シナリオ                   |
| ------------------ | -------------------------- | -------------------------- |
| 発火タイミング     | さわるたびに必ず           | イベント条件成立時のみ     |
| プレイヤー入力待ち | **なし**                   | **あり**                   |
| 分岐               | なし                       | あり                       |
| 部位強制移動       | なし                       | あり                       |
| パラメータボーナス | なし                       | あり（ギミック報酬として） |
| データ形式         | 未定（下記参照）           | Yarn スクリプト            |
| 長さ               | 短い（1〜3アクション程度） | 長い（複数ステップ）       |

### シナリオ中のフィードバックの扱い

シナリオ再生中にさわった場合、フィードバックは再生するか？

> **確認②**: シナリオ中はフィードバックを**抑制する**か、**そのまま出す**か？
>
> - 抑制派：シナリオの没入感を守れる
> - 出す派：「さわったら必ず反応」というゲーム性を保てる

### フィードバックのデータ形式

> **確認③**: フィードバックのデータをどこに持つか？
>
> **案A：Yarn のまま**（最小実装コスト）
>
> - 現在の仕組みをそのまま流用
> - ただし「シナリオではない」概念との乖離がある
>
> **案B：Yarn とは別の軽量形式（YAML / JSON / ScriptableObject）**
>
> - 「フィードバック≠シナリオ」が実装レベルでも明確になる
> - 実装コスト増
>
> **案C：コード直書き（ScriptableObject の Action リスト）**
>
> - Yarn 不要、Unity エディタで管理できる

## シナリオ仕様（Issue #30 / #31）

### 概要

ヒロインが自発的に働きかけ、プレイヤーの入力を受けながら進む一連の流れ。

### シナリオに含まれる機能

#### 1. ヒロインアクション（既存）

```yarn
ヒロインちゃん「ねえっ…こっちも触ってよ…」 #anim:... #expr:...
```

#### 2. プレイヤー入力待ち（新規）

シナリオを一時停止し、プレイヤーの次のおさわりを待つ。

```yarn
<<wait_for_touch>>
```

- どの部位をさわったかが Yarn 変数 `$touched_part` にセットされる
- タイムアウトの有無 → **確認④**

#### 3. 部位強制移動（新規）

プレイヤーの「手」を指定部位に強制的に移動させる。

```yarn
<<force_touch secret_part>>
```

> **確認⑤**: 「強制移動」のUI表現をどうするか？
>
> - **案A: ハイライト** — 指定部位を光らせる（移動強制ではなく誘導）
> - **案B: カーソル強制移動** — 手のアイコンや当たり判定を物理的に移動させる
> - **案C: 部位ロック** — 指定部位以外のさわりを無効化する

#### 4. パラメータボーナス（新規）

```yarn
<<apply_bonus affection 10>>
<<apply_bonus arousal 20>>
```

#### 5. パラメータ条件分岐（Yarn の既存機能で対応可能）

```yarn
<<if $affection > 50 && $development > 30>>
  // 好感度・開発度が高い場合のルート
<<else>>
  // 低い場合のルート
<<endif>>
```

### シナリオ例（Issue #30 コメントのケース）

```yarn
title: chest_to_secret_appeal
tags: part:chest
condition: touch_count:chest:20
priority: 100
---
ヒロインちゃん「ねえっ…こっちも触ってよ…」 #anim:reach_secret #expr:needy
<<force_touch secret_part>>
<<wait_for_touch>>

<<if $touched_part == "secret_part">>
    // 4.a ルート: 言うことを聞いた
    ヒロインちゃん「んっ、きたぁっ♡」 #anim:pleasure #expr:ecstasy
    <<apply_bonus affection 15>>
    <<apply_bonus arousal 20>>
<<elseif $affection > 50>>
    // 4.b ルート: 高好感度で無視
    ヒロインちゃん「やだっ、焦らさないでっ」 #anim:frustrated #expr:pout
    <<force_touch secret_part>>
    <<wait_for_touch>>
    // ...（繰り返しや追加分岐）
<<else>>
    // 4.c ルート: 低好感度で無視
    ヒロインちゃん「……むぅ…」 #anim:sulk #expr:sullen
    <<apply_bonus affection -10>>
    <<apply_bonus arousal -5>>
<<endif>>
===
```

### 4.b ルート問題（「はい」しか選択肢がない問題）について

Issue #30 コメントより：「実質的に『はい』しかないのどうなん？ヒロインの言うことを大人しく聞いてるだけが正解なのもつまんなくね」

> **確認⑥**: 4.b ルートをどう扱うか？
>
> - **案A: そのままにする** — 「強制感」「もどかしさ」もゲーム体験のうち。v0.1 割り切り
> - **案B: 無視し続けたら諦めルート** — 2回無視で 4.c 相当に移行（繰り返し上限を設ける）
> - **案C: 選択肢UIを出す** — 「言うことを聞く / 無視する」ボタン表示（QTE的）

## 回答・確定事項

---

### ① 用語（フィードバック/シナリオ/イベント）で統一してよいか ✅ 確定

フィードバック・シナリオはどちらもイベント経由で発火。

- 簡単な条件でほぼ必ず発火するのがフィードバック（低 priority）
- 特定の条件で発火するのがシナリオ（高 priority）

**設計への影響（重要）**: この設計は既存アーキテクチャと整合し、むしろ実装を簡略化する。

```
低優先度イベント (priority:0)
  条件: touch_count:chest:1  → feedback_chest_default.yarn (フィードバック)

高優先度イベント (priority:100)
  条件: touch_count:chest:20 → chest_to_secret_appeal.yarn (シナリオ)

InteractPart UC は「最高優先度の1つだけ」を発火
→ 通常タッチ: フィードバックが発火
→ 条件成立時: シナリオが発火し、フィードバックは押しのけられる
```

**結論**: `FeedbackID` / `FeedbackRepository` / `FeedbackPlayer` は不要。既存イベントシステムのままフィードバック用 Yarn を追加するだけ。Step 1 の実装量が大幅に減る。

```md
コメント
現在YarnScripts/内にあるyarnファイルはすべてフィードバック側のスクリプトな印象
```

### ② シナリオ中のフィードバックは抑制するか ✅ ほぼ確定

- ヒロインのアクション: **抑制する**
- 擬音等のエフェクト: **抑制しない**（将来対応）

**既存実装で対応済み**: `ScenarioDriver` はシナリオ再生中（`IsPlaying == true`）に低優先度イベントを無視する実装が既にある。つまりアクション抑制は追加実装不要。

**擬音エフェクトについて**: 現在 `Action` に音声の概念がないため、今は「全アクション抑制」の状態。エフェクト非抑制は音声対応時に拡張する。v0.1 では全アクション抑制のまま進める。

### `wait_for_touch` の設計メモ

キャンセルは「強制終了」ではなく Yarn スクリプト側で分岐する。

```yarn
<<wait_for_touch target:secret_part>>
<<if $touched_part == "secret_part">>
    // 4.a ルート: 言うことを聞いた
<<else>>
    <<jump CancelRoute>>  // 4.c ルート相当
<<endif>>

title: CancelRoute
---
ヒロインちゃん「……むぅ…」 #anim:sulk #expr:sullen
<<apply_bonus affection -10>>
===
```

`no_cancel` が必要なシナリオが出てきた場合は Yarn 側の書き方で対応できる（wait 後に `$touched_part` を無視してそのまま進む）。
