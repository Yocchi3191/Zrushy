## アドバイス

### FBの分類を整理する

ヒロインのFBは「レイヤー」で分けて考えると実装しやすい：

| レイヤー     | 内容                       | 揮発性        |
| -------- | ------------------------ | ---------- |
| 即時リアクション | 触ったとき1回だけ出る（ボイス、表情の一瞬変化） | 高（すぐ戻る）    |
| 状態（idle） | 持続する雰囲気（平常/興奮/絶頂後）       | 低（ゆっくり変わる） |
| 永続変化     | 服がずれた、など元に戻らないもの         | 永続         |

即時と状態を混在させると実装がカオスになるので、最初から分けて設計するといい。

### HeroinState VO を作る

`Heroin` エンティティに `HeroinState` みたいな VO を持たせて、idleモードをそこで管理する。
状態変化をイベントとして発行（既存の `EventBus` を流用 or 別建て）すれば、
Unity 側の AnimatorやSpriteSwapper がそれを購読して見た目を変える、という疎結合になる。

```
HeroinState { IdleMode, ArousalLevel }
  ↓ 状態変化
HeroinStateChangedEvent
  ↓ Unity側が購読
AnimationController, ExpressionController, etc.
```

### 即時リアクションと Yarn の関係

「触ったとき一瞬だけセリフ+表情が出る」なら、Yarnシナリオを1行ファイルにして
`InteractPart` から `EventBus` 経由で再生するのが既存の流れに乗る。

Yarnのラインメタデータ（`#expression: surprised` みたいなタグ）は `DialogueRunner` の
`onNodeStart` / `onLinePresentation` で拾えるので、表情や演出の制御に使える。
ただし「メタデータ駆動」は後から仕様が膨らみやすいので、最初はシンプルに
Yarnコマンド（`<<expression surprised>>`）で直接書く方が見通しがいい。

### アセットの粒度

全身アニメは「状態ループ」と「即時リアクション」で用途が違うので別に持つ：

- **ステートループ**：Animatorのステートマシン（平常/興奮/絶頂後）
- **即時リアクション**：Animation Clip を `Animator.Play()` で割り込み再生 → 自動でステートに戻る

表情はスプライト差し替えで十分、全身ポーズはスケルタルかスプライト切替かは絵のスタイル次第。

### 服ずらしについて

おさわりリアクションとして扱うのが正解。
`Part` に `ClothingState`（着用/ずれ/脱げ）を持たせて、インタラクト時に遷移させる。
永続変化なので `IRepository` で保存対象にすることも視野に入れておくといい。

### 優先順位の提案

1. **表情の即時変化**（実装コスト低、FB体感が大きい）
2. **HeroinState + idleアニメ切替**（状態感が出る）
3. **触ったときのボイス**（雰囲気が一気に出る）
4. **服ずらし**（ゲームの核なので早めに）
5. **Yarnとの連携**（シナリオが増えてから）

### 絵を作り直す

レイヤー構造

```
  キャラクター
  ├── 前髪（顔パーツより前面）
  ├── 顔
  │   ├── 頬染め          ← 透明度で興奮度を表現
  │   ├── 口              ← 差し替え（閉じ/開き/喘ぎ）
  │   ├── 目              ← 差し替え（通常/とろん/閉じ）
  │   ├── 眉              ← 差し替え
  │   └── 顔の肌・輪郭   ← 固定
  ├── 身体
  │   ├── 汗・体液        ← 透明度で制御、固定しない
  │   ├── 服（上）        ← 「着用」「ずれ」「脱げ」で別レイヤー or 別ファイル
  │   ├── 下着（上）      ← 同上
  │   ├── 服（下）
  │   ├── 下着（下）
  │   ├── 隠れ部位        ← 服の下に配置、服レイヤーOFFで露出
  │   └── 肌・影          ← 固定
  └── 後ろ髪
```
